## Сессия 2 — Красивая ссылка для парсера через Vercel

### Исследование
- Параллельно запустил 3 агента: Vercel rewrites практики 2026, альтернативы Vercel (custom domain, Cloudflare Tunnel, nip.io), и почему GramJS не работает на Vercel-serverless как backend
- Вывод: бэкенд парсера хостить на Vercel невозможно (Vercel убивает long-lived TCP-сокеты, GramJS требует постоянное MTProto-соединение). Но **Vercel как фронт + reverse-proxy через rewrites** — рабочий путь
- 120-сек жёсткий timeout на proxied requests, WebSocket через rewrites не работает (для нас не критично), `x-vercel-enable-rewrite-caching: 0` обязателен чтобы не закешировать API

### Сделано
- Создан `vercel.json` в корне репы: `outputDirectory: parser/public`, rewrites `/api/*` → backend
- Импортировал репу в Vercel, получил URL `gideon-muiqjp4ta-neuroalex-devs-projects.vercel.app`
- Отключили **Vercel Deployment Protection** (всех ловила Vercel SSO страница, фронт получал HTML вместо JSON и думал что credentials нет)
- Поправил блок `parser.138-16-178-94.nip.io` в Caddyfile (`projects/voice-input/Caddyfile`)
- Сделал `caddy reload`, Caddy выпустил Let's Encrypt сертификат для nip.io домена за пару секунд

### Решения / находки
- **MEMORY.md содержала ошибку про IP:** реальный inbound IP VPS = `138.16.178.94` (видно через `Get-NetIPAddress`), а `165.231.33.74` — это outbound NAT (видно через `api.ipify.org`), на нём сидит чужая машина с сертификатом `*.noooo.win`. DuckDNS A-запись указывала именно на 165 → запросы шли в никуда
- **DuckDNS UI глючил:** показывал что IP обновлён на 138.16.178.94, но authoritative DNS-серверы (ns1/ns2/ns3) держали старый 165 с TTL 1865 сек. Не дождались пропагации — переключились на nip.io
- **nip.io лучше DuckDNS для нашего случая:** wildcard `*.138-16-178-94.nip.io` → IP автоматически, без регистрации, без А-записей, мгновенно. Сертификат через HTTP-01 challenge выпустился сразу
- **Final destination для Vercel:** `https://parser.138-16-178-94.nip.io/api/*`. Юзер видит только Vercel URL, бэкенд скрыт за proxy

## Сессия 3 — Пароль-аутентификация парсера через UI

### Контекст
- Александр попросил «открывать ссылку без добавления токена в URL»
- Решили: пароль через форму, сессия в localStorage, бессрочная (по его выбору)
- Александр сам — «вайбкодер», не программист. Сказал: «Делай всё от начала до конца, не парь мне мозги». Сохранил это в auto-memory feedback_vibe_coder.md

### Сделано
- `parser/lib/web-session.js` — HMAC-SHA256 signed token (`base64url-payload.base64url-signature`), без внешних зависимостей
- `parser/lib/password.js` — scrypt hash/verify (node:crypto, без bcrypt пакета), формат `scrypt$N$r$p$salt$key`
- `parser/scripts/set-password.js` — CLI для recovery: `node parser/scripts/set-password.js <пароль>`
- `parser/server.js` — заменён `requireAuth` (AUTH_TOKEN в query) на `requireSession` (Authorization: Bearer). Добавлены endpoints:
  - `GET /api/auth/needs-setup` — публичный, проверка нужна ли первичная установка пароля
  - `POST /api/auth/setup` — первичная установка (работает только пока hash не задан, 409 после)
  - `POST /api/auth/login` — логин (10 попыток / 5 мин на IP)
  - `POST /api/auth/change-password` — смена пароля (требует currentPassword)
  - `GET /api/auth/me` — проверка валидности сессии
  - `POST /api/auth/logout` — no-op (фронт чистит localStorage)
  - `POST /api/auth/tg-logout` — переименован, был просто logout
- `parser/public/index.html` — три новых экрана: `setup-screen`, `login-screen`, `change-password-screen` + кнопки «Сменить пароль» и «Выйти» в шапке
- `parser/public/app.js` — токен в `localStorage.gideon_parser_session`, шлётся `Authorization: Bearer <token>`. `?token=` в URL больше не используется. Download реализован через fetch+blob (токен не светится в URL). Setup flow + change-password flow
- `parser/public/style.css` — стили для login/setup/change-password экранов и кнопок
- `bot/parser-menu.js` — кнопка «Открыть в браузере» больше не подставляет токен, юзер вводит пароль через форму
- **UI-driven setup**: при первом открытии (если в .env нет LOGIN_PASSWORD_HASH) сайт показывает «Создай пароль» с двумя полями. После — мгновенный вход

### Архитектура auth (для будущих сессий)
- `LOGIN_PASSWORD_HASH` и `SESSION_SECRET` хранятся в `parser/.env`. `SESSION_SECRET` генерируется автоматически при первом запуске, hash — только через UI или CLI
- Loopback bypass в `requireSession` сохранён → бот через `localhost:3000` ходит без пароля как раньше
- Токены бессрочные (нет `exp`). Инвалидация всех сессий = удалить `SESSION_SECRET` из .env, рестарт парсера

### Сделано
- `parser/lib/sessions-manager.js` — управление коллекцией TG-сессий: метаданные в `data/sessions/_meta.json`, файлы `data/sessions/<id>.txt`. Автоматическая миграция со старой `data/session.txt` в `s_migrated` с label «Мой аккаунт» при первом запуске новой версии
- `parser/lib/telegram.js`: `reconfigureClient` (для смены активной без рестарта процесса), `createTempClient` (изолированный клиент для процесса добавления — не затирает активный), `extractSessionString`
- `parser/lib/auth.js`: `sendCodeOnClient`, `signInOnClient` — работают на произвольном клиенте (нужно для temp-flow)
- `parser/server.js`: новые endpoints:
  - `GET /api/sessions` — список + activeId
  - `POST /api/sessions/activate {id}` — переключить активную (отказ 409 если parseInProgress)
  - `PATCH /api/sessions/:id {label}` — переименовать
  - `DELETE /api/sessions/:id` — удалить
  - `POST /api/sessions/add/send-code {phone, label}` — создаёт temp-клиента (TTL 10 мин), шлёт код, возвращает tempId
  - `POST /api/sessions/add/sign-in {tempId, code, password?, label, activate}` — завершает auth, сохраняет StringSession + metadata
- `/api/auth/status` расширен полями `hasActiveSession`, `activeSession`
- `/api/auth/tg-logout` теперь удаляет активную сессию из менеджера, переключается на следующую если есть
- Frontend: кнопка `🔄 TG-аккаунты` в шапке. Новые экраны:
  - `sessions-screen` — список с кнопками Использовать / Переименовать / Удалить + кнопка `+ Добавить TG-аккаунт`
  - `add-tg-screen` — форма label → phone → code → 2FA. Заменяет старый auth-screen
- `loadAfterLogin` теперь: если нет activeSession → автоматически открывает add-tg-screen (firstRun=true)

### Архитектура
- Старая `data/session.txt` после миграции переименовывается в `data/session.txt.bak`
- Конкретно для нашего случая в каждый момент времени работает один GramJS-клиент (активная сессия). При смене активной — disconnect старого + новый клиент с другой сессией
- Temp-клиенты для добавления (TTL 10 мин) живут параллельно активному, не конфликтуют
